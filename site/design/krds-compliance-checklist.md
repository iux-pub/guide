KRDS의 원칙, 스타일, 컴포넌트, 기본 패턴, 서비스 패턴을 프로젝트 맥락에 맞게 적용했는지 점검하기 위한 체크리스트다. 이 문서는 KRDS 화면을 픽셀 단위로 복제했는지를 확인하는 문서가 아니라, KRDS 세부 지침의 용례·구조·사용성·상호작용·접근성 기준을 훼손하지 않았는지 검수하기 위한 기준이다.

먼저 [사이트 유형 판정 기준](/design/project-profiles/)에 따라 일반사이트, 공공서비스, 공공기관, CMS·관리자, 커머스·예약 중 하나로 판정한다. 공식 배너, 정부 상징, 운영기관 식별자, 공공 푸터 필수 링크는 모든 사이트의 기본 생성값이 아니며, 공공서비스 또는 요구사항이 확인된 공공기관 프로젝트에서만 조건부로 적용한다. 일반사이트, CMS·관리자, 커머스·예약에서는 해당 항목을 `N/A`로 처리한다.

## 판정 기준

| 판정 | 의미 |
|------|------|
| PASS | KRDS 지침을 충족하며 구현·증빙이 가능하다 |
| 조건부 PASS | 서비스 맥락상 변형했으나 사유와 대체 기준을 문서화했다 |
| FAIL | KRDS 필수 지침, 접근성, 사용성, 상호작용 기준을 충족하지 못한다 |
| N/A | 해당 화면·기능에 적용 대상이 아니다 |

## 0. 적용 범위 정의

- [ ] 사이트 유형을 일반사이트, 공공서비스, 공공기관, CMS·관리자, 커머스·예약 중 하나로 판정했다
- [ ] 정부 상징/공식 배너/운영기관 식별자 적용 여부를 과업지시서 또는 기관 정책으로 확인했다
- [ ] RFP 또는 과업지시서에서 요구하는 KRDS 준수 범위를 확인했다
- [ ] 점검 대상 화면, 기능, 사용자 여정을 목록화했다
- [ ] 해당 서비스에 적용되는 서비스 패턴을 식별했다
- [ ] 서비스 패턴이 없을 경우 유사 서비스 패턴 또는 관련 기본 패턴을 지정했다
- [ ] 신규 구축, 개편, 일부 개선, 운영 개선 중 어떤 적용 상황인지 정리했다
- [ ] 필수, 권장, 우수 중 이번 사업에서 목표로 하는 준수 수준을 정했다
- [ ] 적용 제외 항목은 N/A 사유를 남겼다

## 1. 원칙 점검

- [ ] 사용자의 핵심 과업을 기준으로 화면 흐름을 설계했다
- [ ] 사용자가 현재 위치, 진행 상태, 다음 행동을 이해할 수 있다
- [ ] 불필요한 절차나 중복 입력을 줄였다
- [ ] 오류 발생 시 원인과 해결 방법을 명확히 제공한다
- [ ] PC, 모바일, 보조 기술 사용자를 포함한 다양한 이용 상황을 고려했다
- [ ] 시각적 표현보다 과업 완료와 정보 이해를 우선했다
- [ ] 서비스 특성상 KRDS와 다르게 설계한 항목은 사유를 문서화했다

## 2. 스타일 점검

### 색상

- [ ] 주요 색상은 KRDS 또는 INFOMIND 토큰에 매핑된다
- [ ] 본문, 보조 텍스트, 비활성, 링크, 오류, 성공, 경고 색상이 의미 토큰으로 구분된다
- [ ] 색상만으로 상태나 정보를 전달하지 않는다
- [ ] 일반 텍스트 대비는 4.5:1 이상이다
- [ ] 큰 텍스트와 UI 경계 대비는 3:1 이상이다
- [ ] 브랜드 색상 오버라이드는 접근성 대비를 통과한다

### 글꼴과 타이포그래피

- [ ] 기본 글꼴은 프로젝트 기준 글꼴을 사용한다
- [ ] 제목, 본문, 캡션, 버튼, 폼 레이블의 계층이 명확하다
- [ ] 반복되는 타이포그래피 값은 토큰 또는 공통 클래스로 정리했다
- [ ] 정보 밀도가 높은 CMS·관리자 화면은 가독성과 스캔성을 우선해 조정했다
- [ ] 줄간격과 문단 간격이 읽기 흐름을 방해하지 않는다

### 형태와 레이아웃

- [ ] 컨테이너, 그리드, 여백은 일관된 기준으로 적용했다
- [ ] 화면 큰 영역은 기존 인포마인드 패턴인 `.skip-to-content`, `header#header`, `main#main`, `footer#footer` 구조를 유지한다
- [ ] `main` 내부는 `section > .container` 단위로 구성한다
- [ ] 각 `section`은 heading 또는 `aria-labelledby`/`aria-label`로 접근 가능한 이름을 제공한다
- [ ] 반복되는 radius, shadow, border 값은 토큰 또는 공통 패턴으로 관리한다
- [ ] 컴포넌트 간 간격은 화면 맥락에 맞지만 무작위 값처럼 보이지 않는다
- [ ] 모바일, 태블릿, PC에서 정보 우선순위가 유지된다

### 아이콘과 이미지

- [ ] 아이콘은 의미가 명확하고 동일 기능에 동일 아이콘을 사용한다
- [ ] 장식 아이콘과 정보 전달 아이콘을 구분했다
- [ ] 아이콘 버튼은 접근 가능한 이름을 가진다
- [ ] 이미지는 목적에 맞는 대체 텍스트를 가진다
- [ ] 기관, 서비스, 브랜드 이미지 사용 시 왜곡이나 과도한 장식이 없다

## 3. 컴포넌트 점검

- [ ] KRDS 컴포넌트 카탈로그에서 대응 가능한 컴포넌트를 먼저 확인했다
- [ ] 공식 배너, 정부 상징, 운영기관 식별자, 공공 푸터는 적용 대상 프로젝트에서만 생성했다
- [ ] 민간·사내·일반 CMS 프로젝트에서는 정부 아이덴티티 요소를 생성하지 않고 N/A 사유를 기록했다
- [ ] 버튼, 링크, 입력, 선택, 탐색, 피드백 요소는 용례에 맞게 선택했다
- [ ] 컴포넌트 구조가 KRDS의 의미 구조와 상호작용 의도를 벗어나지 않는다
- [ ] Default, Hover, Focus, Active, Disabled 상태를 제공한다
- [ ] 오류, 성공, 경고, 정보 상태가 시각적 표현과 텍스트로 함께 제공된다
- [ ] 인터랙티브 요소는 `<button>`, `<a>`, `<input>`, `<select>` 등 시맨틱 HTML을 우선 사용한다
- [ ] 모달, 탭, 아코디언, 툴팁, 캐러셀 등 복합 위젯은 필수 ARIA와 키보드 조작을 제공한다
- [ ] 모바일 터치 영역은 최소 44x44px 이상이다
- [ ] 컴포넌트 변형이 필요한 경우 일회성 변형인지 공통 컴포넌트 후보인지 판단했다
- [ ] KRDS와 다른 표현을 사용한 경우 사용자 과업, 정보 밀도, 기술 제약 중 어떤 이유인지 기록했다

## 4. 기본 패턴 점검

### 입력폼

- [ ] 레이블, 도움말, 입력 필드, 오류 메시지의 관계가 명확하다
- [ ] 필수/선택 입력 여부를 사용자가 알 수 있다
- [ ] 입력 예시와 형식 제한을 사전에 제공한다
- [ ] 오류 메시지는 발생 위치 근처에 제공하고 해결 방법을 포함한다
- [ ] 첨부파일, 날짜, 검색, 필터 등 입력 유형별 사용성 기준을 확인했다

### 탐색과 목록

- [ ] 사용자가 현재 위치와 상위 구조를 이해할 수 있다
- [ ] 메인 메뉴, 브레드크럼, 페이지네이션, 탭 등 탐색 컴포넌트가 용례에 맞다
- [ ] 목록은 정렬, 필터, 검색, 페이지 이동 방식이 명확하다
- [ ] 상세 화면으로 이동하는 링크 텍스트가 목적지를 설명한다
- [ ] 빈 결과, 로딩, 오류 상태를 제공한다

### 도움, 확인, 오류, 피드백

- [ ] 사용자가 막히기 전에 필요한 도움말을 제공한다
- [ ] 중요한 작업 전 확인 절차를 제공한다
- [ ] 삭제, 제출, 결제 등 되돌리기 어려운 행동은 결과를 명확히 안내한다
- [ ] 작업 완료 후 성공 여부와 다음 행동을 안내한다
- [ ] 시스템 오류는 사용자가 취할 수 있는 조치와 문의 경로를 제공한다

## 5. 서비스 패턴 점검

- [ ] 해당 서비스의 핵심 사용자 여정을 정의했다
- [ ] KRDS 서비스 패턴 중 직접 대응되는 패턴을 확인했다
- [ ] 직접 대응되는 패턴이 없으면 유사 과업 패턴을 근거로 지정했다
- [ ] 서비스 패턴의 필수 수준 지침을 우선 반영했다
- [ ] 권장·우수 수준 항목 중 사업 범위에 포함할 항목을 선별했다
- [ ] 사용자 여정별 시작, 입력, 확인, 완료, 오류, 재시도 흐름을 점검했다
- [ ] 표준 프로토타입을 참고하되 서비스 특성과 정보 구조에 맞게 조정했다
- [ ] 조정한 항목은 “KRDS 의도 유지 + 프로젝트 맥락상 변형”으로 설명 가능하다

## 6. 구현 점검

- [ ] 색상은 `var(--color-*)` 토큰을 사용한다
- [ ] 간격, 크기, 타이포그래피는 반복 패턴에 한해 토큰화를 우선 검토했다
- [ ] 프로젝트 고유 레이아웃 값은 px/rem 사용 사유가 명확하다
- [ ] CSS는 표준 CSS nesting과 Tailwind v4 `@apply`를 사용한다
- [ ] SCSS 전용 문법을 사용하지 않는다
- [ ] BEM은 `5-objects`, `6-components` 레이어의 공통 패턴에 적용한다
- [ ] 상태 클래스는 시각적 단어가 아니라 의미적 modifier를 사용한다
- [ ] `:focus { outline: none }`처럼 접근성을 제거하는 코드는 없다
- [ ] 본문 건너뛰기 링크를 제공한다
- [ ] `npm run check`, `npm run lint`, `npm run test:a11y` 결과를 확인했다

## 7. 산출물과 증빙

- [ ] KRDS 적용 범위 표를 작성했다
- [ ] 화면별 체크리스트 결과를 남겼다
- [ ] PASS, 조건부 PASS, FAIL, N/A 판정 결과를 요약했다
- [ ] 조건부 PASS 항목은 변형 사유와 대체 기준을 기록했다
- [ ] FAIL 항목은 수정 계획과 담당자를 기록했다
- [ ] 주요 컴포넌트의 상태 화면을 캡처했다
- [ ] 모바일, 태블릿, PC 대표 화면 캡처를 확보했다
- [ ] 접근성 검사 결과 또는 수동 점검 기록을 남겼다
- [ ] 최종 납품 전 이 체크리스트를 기준으로 재점검했다
