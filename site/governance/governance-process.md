가이드 시스템의 토큰과 컴포넌트를 변경하려면 정해진 승인 프로세스를 따라야 한다. 이를 통해 변경의 영향을 사전에 파악하고, 팀 전체가 합의한 상태에서 반영한다.

## 토큰 변경 프로세스

디자인 토큰(색상, 간격, 타이포그래피 등)의 추가, 수정, 삭제 시 다음 절차를 따른다.

### 1. 제안

- GitHub 이슈로 변경 요청을 등록한다
- 필수 기재 항목:
  - 변경 대상 토큰 이름
  - 변경 사유 (왜 필요한지)
  - 변경 전/후 값
  - 영향 범위 (사용 중인 컴포넌트, 프로젝트 목록)

### 2. 팀 리뷰

- 영향받는 컴포넌트와 프로젝트 목록을 확인한다
- 기존 프로젝트에 미치는 영향을 분석한다
- 대안이 있는지 검토한다

### 3. 승인

- **토큰 추가(MINOR)**: 팀 리드 또는 2인 이상 동의
- **토큰 수정/삭제(MAJOR)**: 팀 전체 합의 필요
- 승인 결과를 이슈에 기록한다

### 4. 반영

1. 변경 사항을 구현한 PR을 생성한다
2. 코드 리뷰를 거친다
3. 머지 후 CHANGELOG를 업데이트한다
4. 영향받는 문서 페이지를 수정한다

## 컴포넌트 추가 프로세스

새로운 컴포넌트를 가이드 시스템에 추가할 때 다음 절차를 따른다.

### 1. 제안

- GitHub 이슈로 컴포넌트 제안을 등록한다
- **유즈케이스 3개 이상** 제시 필수 (실제 프로젝트에서 필요한 사례)
- 기존 컴포넌트로 대체 불가능한 이유를 설명한다

### 2. 디자인 리뷰

- 피그마 컨벤션 준수 여부를 확인한다
  - 컴포넌트 네이밍 규칙
  - Variable(토큰) 사용
  - Auto Layout 규칙
- 반응형 동작을 정의한다 (모바일/태블릿/PC)

### 3. 구현

- HTML 마크업 + SCSS 스타일 스니펫을 작성한다
- BEM 네이밍 컨벤션을 준수한다
- 디자인 토큰(CSS Custom Properties)을 사용한다
- `src/snippets/`에 마크다운 스니펫 문서를 작성한다
- `src/scss/6-components/`에 SCSS 파일을 추가한다

### 4. QA

- **접근성**: pa11y 검사 통과 (KWCAG/WCAG AA)
- **크로스 브라우저**: Chrome, Edge, Safari 최신 + IE11(공공기관)
- **반응형**: 모바일(360px~), 태블릿(768px~), PC(1280px~)
- **Stylelint**: BEM 패턴 검증 통과

### 5. stable 승격

- [컴포넌트 라이프사이클](/governance/lifecycle/) 기준을 충족하면 stable로 승격한다
- 최초 추가 시 experimental로 시작하고, 검증을 거쳐 beta -> stable로 승격한다

## 컴포넌트 변경 프로세스

기존 컴포넌트의 마크업, 스타일, 접근성 속성을 변경할 때 다음 절차를 따른다.

### 변형(variant) 추가

1. GitHub 이슈 등록 (유즈케이스, 변경 내용)
2. PR 생성 + 코드 리뷰
3. CHANGELOG에 Added로 기록

### 기존 마크업/스타일 변경

1. GitHub 이슈 등록 (변경 사유, 영향 범위)
2. 팀 리뷰 (2인 이상 동의)
3. PR 생성 + 코드 리뷰
4. CHANGELOG에 Changed로 기록

### 컴포넌트 삭제

1. deprecated 라벨링 (6개월 유예)
2. 대체 컴포넌트 안내 및 마이그레이션 가이드 제공
3. MAJOR 버전에서만 최종 삭제
4. CHANGELOG에 Removed로 기록

## 긴급 변경

다음 경우에는 리뷰를 **병행**하여 즉시 반영할 수 있다:

- **접근성 결함**: KWCAG/WCAG AA 위반 사항 발견 시
- **보안 이슈**: 보안 취약점이 발견된 경우

긴급 변경도 반드시 이슈와 PR을 생성하되, 리뷰와 머지를 동시에 진행한다. 사후에 팀 전체에 공유하고 CHANGELOG를 업데이트한다.
