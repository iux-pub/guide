# Architecture Patterns

**Domain:** Design System Documentation & Publishing Guide
**Researched:** 2026-03-25

## Recommended Architecture

This project has two deliverables: (1) a documentation site and (2) a reusable SCSS/HTML template system.

### Component Boundaries

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|
| Token Source (JSON) | Single source of truth for design values | Style Dictionary config |
| Style Dictionary | Transforms tokens to SCSS vars + CSS custom props | SCSS Settings layer, documentation pages |
| SCSS Template System | The actual deliverable: reusable styles | Compiled CSS output, documentation site |
| Eleventy Site | Renders documentation pages with live previews | SCSS compiled output, token data |
| Stylelint Config | Enforces BEM + SCSS conventions | Developer workflow, CI |
| pa11y-ci | Validates accessibility of built pages | Built documentation site |

### Data Flow

```
tokens/*.json
    |
    v
Style Dictionary (build:tokens)
    |
    +----> src/scss/1-settings/_tokens.scss    (SCSS variables)
    +----> src/scss/1-settings/_custom-props.scss (CSS custom properties)
    +----> src/site/_data/tokens.json          (for documentation display)
    |
    v
SCSS Compilation (build:scss)
    |
    +----> dist/css/style.css
    |
    v
Eleventy Build (build:site)
    |
    +----> dist/index.html, dist/components/*.html, etc.
    |
    v
pa11y-ci (test:a11y)
    |
    +----> Accessibility report
```

## Patterns to Follow

### Pattern 1: ITCSS Layer Ordering

**What:** Import SCSS partials in strict specificity order
**When:** Always. This is the core architecture decision.
**Example:**
```scss
// style.scss - Main entry point
// ITCSS layers: low specificity -> high specificity

// 1. Settings - tokens, variables (no CSS output)
@use '1-settings/tokens' as *;
@use '1-settings/breakpoints' as *;

// 2. Tools - mixins, functions (no CSS output)
@use '2-tools/mixins' as *;
@use '2-tools/functions' as *;

// 3. Generic - reset, normalize (first CSS output)
@use '3-generic/normalize';
@use '3-generic/box-sizing';

// 4. Elements - bare HTML elements
@use '4-elements/headings';
@use '4-elements/links';
@use '4-elements/tables';

// 5. Objects - layout patterns
@use '5-objects/container';
@use '5-objects/grid';

// 6. Components - BEM components
@use '6-components/btn';
@use '6-components/card';
@use '6-components/form';

// 7. Utilities - helper classes (highest specificity)
@use '7-utilities/sr-only';
@use '7-utilities/visibility';
```

### Pattern 2: BEM Component Structure

**What:** Each component file follows consistent BEM structure
**When:** Every component in layer 6
**Example:**
```scss
// _btn.scss
.btn {
  // Block styles
  display: inline-flex;
  align-items: center;
  justify-content: center;
  padding: rem(8px) rem(16px);
  font-size: rem(14px);
  line-height: 1.4;
  border: 1px solid transparent;
  border-radius: rem(4px);
  transition: all 0.4s;
  cursor: pointer;

  // Elements
  &__icon {
    margin-right: rem(8px);
  }

  &__text {
    // text-specific styles
  }

  // Modifiers
  &--primary {
    background-color: var(--color-primary);
    color: var(--color-white);
  }

  &--secondary {
    background-color: var(--color-secondary);
    color: var(--color-white);
  }

  &--large {
    padding: rem(12px) rem(24px);
    font-size: rem(16px);
  }

  &--small {
    padding: rem(4px) rem(12px);
    font-size: rem(12px);
  }
}
```

### Pattern 3: Design Token to SCSS Pipeline

**What:** JSON tokens generate both SCSS variables and CSS custom properties
**When:** Whenever design values change
**Example:**
```json
// tokens/color.json (DTCG format)
{
  "color": {
    "primary": {
      "$value": "#1886d8",
      "$type": "color",
      "$description": "Main brand color, project-specific override"
    },
    "black": {
      "$value": "#222222",
      "$type": "color"
    },
    "gray": {
      "900": { "$value": "#404040", "$type": "color" },
      "600": { "$value": "#666666", "$type": "color" },
      "400": { "$value": "#9ca3af", "$type": "color" },
      "200": { "$value": "#cccccc", "$type": "color" },
      "100": { "$value": "#f0f0f0", "$type": "color" }
    }
  }
}
```

### Pattern 4: Component Documentation Page (Eleventy + Nunjucks)

**What:** Each component gets a documentation page with live preview + code
**When:** Every component in the guide
**Example:**
```html
<!-- src/site/pages/components/btn.njk -->
---
layout: layouts/component.njk
title: Button
tags: component
---

<h2>Button</h2>
<p>Standard button component with multiple variants.</p>

<h3>Default</h3>
<div class="preview">
  <button class="btn btn--primary">
    <span class="btn__text">Primary Button</span>
  </button>
</div>

<h3>Code</h3>
{% highlight "html" %}
<button class="btn btn--primary">
  <span class="btn__text">Primary Button</span>
</button>
{% endhighlight %}

<h3>Accessibility</h3>
<ul>
  <li>Use <code>&lt;button&gt;</code> for actions, <code>&lt;a&gt;</code> for navigation</li>
  <li>Always provide visible text or <code>aria-label</code></li>
  <li>Ensure 4.5:1 color contrast ratio</li>
</ul>
```

## Anti-Patterns to Avoid

### Anti-Pattern 1: @import instead of @use/@forward

**What:** Using deprecated `@import` for SCSS partials
**Why bad:** @import is deprecated in Dart Sass. Will be removed in future versions. Causes global namespace pollution.
**Instead:** Use `@use` with namespaces and `@forward` for re-exporting.

### Anti-Pattern 2: Nesting beyond 3 levels

**What:** Deep SCSS nesting like `.block__element .another .thing { ... }`
**Why bad:** Creates high-specificity selectors, hard to override, defeats BEM purpose.
**Instead:** Keep nesting to Block > Element > Modifier (max 3 levels). Flatten when possible.

### Anti-Pattern 3: Component styles in Objects layer

**What:** Putting visual styles (colors, borders) in ITCSS Objects layer
**Why bad:** Objects should be structural only (layout). Mixing concerns breaks the inverted triangle.
**Instead:** Objects = layout (grid, container). Components = visual (btn, card).

### Anti-Pattern 4: Hardcoded values instead of tokens

**What:** Writing `color: #1886d8` directly in component files
**Why bad:** Cannot be overridden per-project. No single source of truth.
**Instead:** Use `var(--color-primary)` or SCSS variable from tokens.

## Scalability Considerations

| Concern | 1 Project | 5 Projects | 20+ Projects |
|---------|-----------|------------|--------------|
| Token overrides | Edit JSON directly | Project-specific token overrides via CSS custom properties | Token theming system with project configs |
| Component count | ~10 components | ~20 components | Component categorization (atoms/molecules) |
| Documentation | Single Eleventy site | Same site, versioned | Consider separate npm package for SCSS |
| Build time | <5s | <5s (same build) | Still <10s (Eleventy is fast) |
